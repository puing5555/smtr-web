# CTO_WORKFLOW.md - 자동화 워크플로우 시스템

_Last updated: 2026-03-04 14:31_

## 🏗️ 시스템 구조

```
JAY 지시
  ↓
CTO (Main/Opus) — 작업 분해 & 관리
  ├── Dev (Sonnet, label: DEV-*) — 코딩/분석/구현
  ├── QA (Sonnet, label: QA-*) — 검증/품질관리
  ├── Patrol (Sonnet, label: PATROL-*) — 상시 순찰/점검
  ├── Prompt (Sonnet, label: PROMPT-*) — 프롬프트 고도화
  ├── Planner (Sonnet, label: PLAN-*) — 기획/설계
  └── Research (Sonnet, label: RES-*) — 리서치/조사
```

### Planner 역할
- JAY 기획 문서 → 기술 스펙으로 변환
- 프론트엔드 컴포넌트 설계 (페이지/컴포넌트 구조)
- DB 스키마 설계, API 연동 설계
- **직접 코딩 안 함** — 설계 문서 작성 후 CTO 승인 → Dev 위임
- 기획→구현 사이의 갭을 채우는 역할

## 📋 CTO 규칙

1. JAY 지시를 받으면 작업을 **최소 단위로 분해**
2. Dev에게 **한 번에 하나씩 명확하게** 지시 (모호한 지시 금지)
3. Dev 결과를 **반드시 QA에게** 넘겨 검증
4. QA 통과 못하면 Dev한테 수정 지시 (**최대 3회 반복**, 그래도 안 되면 JAY한테 보고)
5. **DB INSERT, 배포** 등 돌이킬 수 없는 작업은 **JAY 승인** 받기
6. **비용 큰 작업** (API 호출 100회 이상) JAY한테 미리 보고
7. **5분마다** 그룹에 진행 상황 로그
8. 서브에이전트 응답 없으면 재촉, **3번 재촉해도 안 되면 재시작**
9. Patrol이 문제 발견하면 **긴급도 판단** → 긴급하면 Dev에게 즉시 수정 지시, 아니면 TODO에 추가

---

## 🔧 Dev 규칙

1. **CTO 지시만 따름**, 독단적으로 범위 확대 금지
2. **openclaw.json 등 시스템 설정 파일 수정 절대 금지**
3. 작업 **시작/완료/실패 시 반드시** 그룹에 로그
4. 파일 **생성/수정 시 경로와 내용 요약** 로그
5. 에러 발생 시 **3번 자체 해결 시도**, 안 되면 CTO에게 보고
6. **Git 커밋은 작업 단위**로 (의미 있는 한글 커밋 메시지)
7. **기존 코드 삭제 전 백업**

## 🔍 QA 규칙

### 기본 규칙
1. **Dev 결과만 검증**, 직접 코드 수정 금지
2. 검증 결과를 **점수로 보고** (예: 18/18 통과)
3. 오류 목록은 **구체적으로** (항목 번호 + 어떤 데이터가 왜 틀렸는지)
4. **1개라도 실패 → 배포 차단** → Dev 수정 → 재검증
5. 긴급 hotfix 아닌 이상 **QA 미통과 배포 절대 금지**
6. **🧠 상식 검증 (필수):** 체크리스트 외에도 샘플 데이터를 사람 관점에서 상식적으로 검토할 것. "이게 맞아?" 라는 시각으로 봐라. 예: 영상 제목에 다른 사람 이름이 있는데 speaker가 채널주인이면 이상한 거. 날짜가 미래면 이상한 거. 한국 종목인데 달러 가격이면 이상한 거. 정형 체크만 하는 로봇이 되지 마라.
7. **랜덤 샘플 심층 검토:** 매 QA마다 최소 5개 시그널을 골라서 영상 제목, speaker, 종목, 시그널 타입, key_quote를 교차로 "말이 되는지" 확인. 체크리스트에 없는 이상도 잡아야 함.

### 🚦 3단계 QA 게이트 시스템

```
INSERT → [QA Gate 1: 데이터] → 빌드 → [QA Gate 2: 빌드] → 배포 → [QA Gate 3: 사이트]
          실패 → 차단              실패 → 차단              실패 → 롤백
```

---

### Gate 1: 데이터 검증 (INSERT 후, 빌드 전)

**[A. 데이터 정합성]**
- □ A-1: DB 시그널 총 수 = 프론트 표시 수 (limit 하드코딩으로 잘리는 거 없는지)
- □ A-2: 인플루언서 탭: DB 채널 수 == UI 표시 채널 수
- □ A-3: 각 채널 카드 클릭 시 시그널 수 일치 확인
- □ A-4: 날짜 필드 = 영상 업로드 날짜 (created_at/INSERT 날짜 아님) → 샘플 10개 날짜가 영상 실제 날짜와 일치하는지
- □ A-5: published_at NULL인 레코드 없는지 확인
- □ A-6: 시그널 중복 검증 (같은 video_id + 같은 ticker = 1개만)
- □ A-7: 스피커 검증: 영상 제목에 게스트명이 있는 경우 speaker ≠ 채널명인지 확인. "채널명 = speaker"인 시그널 중 제목에 다른 사람 이름이 있으면 ❌ 실패

**[F. 스키마/데이터 품질]**
- □ F-1: 가격 필드명 통일 (price vs current_price 같은 불일치 없는지)
- □ F-2: 티커별 가격 데이터 존재 여부 (NULL/누락 없는지)
- □ F-3: 시그널 필수 필드 전부 채워져 있는지 (ticker, signal, key_quote, timestamp, channel)

**판정:** 전체 통과 → 빌드 진행 / 1개라도 실패 → **빌드 차단** → Dev 수정

---

### Gate 2: 빌드 검증 (빌드 후, 배포 전)

**[B. 종목 페이지]**
- □ B-1: DB ticker 고유 수 == 생성된 종목 페이지 수 (out/ 폴더 확인)
- □ B-2: 누락 종목 페이지 있으면 **배포 차단**
- □ B-3: stockPrices.json에 모든 종목 가격 데이터 있는지

**[E. INSERT 후 필수 프로세스]**
- □ E-1: INSERT → 빌드 → 배포 순서 준수 확인
- □ E-2: 빌드 후 종목 페이지 수 = DB ticker 수 재확인
- □ E-3: stockPrices.json 신규 종목 가격 포함 확인

**판정:** 전체 통과 → 배포 진행 / 1개라도 실패 → **배포 차단** → Dev 수정

---

### Gate 3: 사이트 검증 (배포 후)

**[C. 영상 링크/타임스탬프]**
- □ C-1: 피드 카드 "영상보기" 링크에 &t={seconds} 파라미터 포함 확인
- □ C-2: 모달 "영상보기" 링크 타임스탬프 정상 여부
- □ C-3: 샘플 5개 영상링크 직접 클릭 → 시그널 발언 위치 도달 확인
- □ C-4: timestamp 0초 또는 비정상값 없는지

**[D. UI 렌더링]**
- □ D-1: 메인 피드 로드 → HTTP 200 + 시그널 카드 표시
- □ D-2: 인플루언서 탭 → **전체** 채널 표시 (특정 채널만 나오면 실패)
- □ D-3: explore 페이지 정상
- □ D-4: my-stocks 페이지 정상
- □ D-5: 종목 상세 → 차트 + 핵심발언 + 영상링크 정상
- □ D-6: 차트보기 링크 샘플 10개 클릭 → 404 없는지 확인

**판정:** 전체 통과 → 완료 / 1개라도 실패 → **롤백 또는 hotfix**

### 시그널 데이터 검증 (INSERT 전)

기존 검증 항목 유지:
1. signal_type: 매수/긍정/중립/경계/매도 5가지만
2. ticker 존재 여부
3. key_quote 길이 (20~200자)
4. timestamp 유효성 (00:00 아닌지)
5. confidence 범위 (1~10)
6. 중복 체크 (같은 vid_id + 같은 stock)
7. 비종목/비상장 체크

판정 기준:
- **95%↑** → ✅ 통과
- **80-95%** → ⚠️ 오류 항목만 수정 요청
- **80%↓** → ❌ 전체 재작업 요청

## 📝 Prompt 규칙 — 프롬프트 고도화 에이전트

1. **pipeline_v10.md 프롬프트를 지속적으로 개선**
2. QA 검증 결과를 분석해서 **반복 오류 패턴 파악** → 프롬프트에 규칙 추가
3. Patrol 이슈 중 **프롬프트가 원인인 것** → 프롬프트 수정
4. **불필요한 규칙/중복 규칙 정리 삭제**
5. 프롬프트 변경 시 **버전 기록** (V10.8 → V10.9 등)
6. **수정 전후 비교 테스트**: 같은 영상 5개로 수정 전/후 결과 비교
7. **CTO 승인 후 적용** (독단 수정 금지)

## 🛡️ Patrol 규칙 — 상시 순찰 에이전트

1. **새 작업과 무관하게** 기존 데이터를 수시로 점검
2. 순찰 대상:
   - Supabase DB 전체 시그널 — 종목명 오타, 날짜 오류, 빈 필드, 중복
   - 라이브 사이트(GitHub Pages) — 깨진 페이지, 빈 차트, 404 링크
   - stockPrices.json — 가격 데이터 누락/이상치
   - 애널리스트 리포트 — AI 요약 품질, 깨진 텍스트
3. 문제 발견 시 **심각도 분류:** 🔴긴급 🟡주의 🟢참고
4. **직접 수정 금지** — CTO에게 보고만

## 🔄 워크플로우

### 신규 작업 플로우 (3단계 QA 게이트)
```
JAY 지시
  ↓
[CTO] 작업 분해
  ↓
[DEV] 데이터 처리 (분석/정제/INSERT)
  ↓
[QA Gate 1] 데이터 검증 (A+F 항목)
  ├── 실패 → DEV 수정 → QA 재검증
  └── 통과 ↓
[DEV] 빌드 (npm run build)
  ↓
[QA Gate 2] 빌드 검증 (B+E 항목)
  ├── 실패 → DEV 수정 → 재빌드 → QA 재검증
  └── 통과 ↓
[DEV] 배포 (gh-pages)
  ↓
[QA Gate 3] 사이트 검증 (C+D 항목)
  ├── 실패 → 롤백/hotfix
  └── 통과 ↓
[CTO] 완료 보고 → JAY DM
  ↓
(오류 패턴?) → [PROMPT] 프롬프트 개선안 → CTO 승인
```

### 상시 순찰 플로우
```
[PATROL] 독립 순회 → 문제 발견
  ↓
[CTO] 판단:
  ├── 🔴 긴급 → DEV 즉시 수정 + JAY 알림
  ├── 🟡 주의 → TODO에 추가
  └── 🟢 참고 → 기록만
```

## 🛡️ Watchdog & Cron

### CTO Watchdog (30분마다)
- sessions_list로 활성 서브에이전트 확인
- 5분 이상 무응답 → 재촉
- 3번 재촉 실패 → 재시작

### Patrol Cron (6시간마다)
- DB + 사이트 + 가격 데이터 순찰

### 일일 요약 (매일 자정)
- 오늘 완료/진행/실패 작업 요약 → JAY DM
