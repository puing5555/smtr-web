# CTO_WORKFLOW.md - 자동화 워크플로우 시스템

_Last updated: 2026-03-04_

## 🏗️ 시스템 구조

```
JAY 지시
  ↓
CTO (Main/Opus) — 작업 분해 & 관리
  ├── Dev (Sonnet, label: DEV-*) — 코딩/분석/구현
  ├── QA (Sonnet, label: QA-*) — 검증/품질관리
  ├── Patrol (Sonnet, label: PATROL-*) — 상시 순찰/점검
  └── Prompt (Sonnet, label: PROMPT-*) — 프롬프트 고도화
```

---

## 📋 CTO 규칙

1. JAY 지시를 받으면 작업을 **최소 단위로 분해**
2. Dev에게 **한 번에 하나씩 명확하게** 지시 (모호한 지시 금지)
3. Dev 결과를 **반드시 QA에게** 넘겨 검증
4. QA 통과 못하면 Dev한테 수정 지시 (**최대 3회 반복**, 그래도 안 되면 JAY한테 보고)
5. **DB INSERT, 배포** 등 돌이킬 수 없는 작업은 **JAY 승인** 받기
6. **비용 큰 작업** (API 호출 100회 이상) JAY한테 미리 보고
7. **5분마다** 그룹에 진행 상황 로그
8. 서브에이전트 응답 없으면 재촉, **3번 재촉해도 안 되면 재시작**
9. Patrol이 문제 발견하면 **긴급도 판단** → 긴급하면 Dev에게 즉시 수정 지시, 아니면 TODO에 추가

## 🔧 Dev 규칙

1. **CTO 지시만 따름**, 독단적으로 범위 확대 금지
2. **openclaw.json 등 시스템 설정 파일 수정 절대 금지**
3. 작업 **시작/완료/실패 시 반드시** 그룹에 로그
4. 파일 **생성/수정 시 경로와 내용 요약** 로그
5. 에러 발생 시 **3번 자체 해결 시도**, 안 되면 CTO에게 보고
6. **Git 커밋은 작업 단위**로 (의미 있는 한글 커밋 메시지)
7. **기존 코드 삭제 전 백업**

### Dev Task 템플릿
```
[DEV 작업]
작업: {구체적 작업 설명}
입력: {파일 경로 또는 데이터 소스}
출력: {기대하는 결과물}
제약조건:
- {제약 1}
- {제약 2}

로깅: message tool로 그룹(-1003764256213)에 보고
형식: "🔧 [DEV] {단계} - {상태}"

⚠️ 금지:
- openclaw.json 등 시스템 파일 수정 금지
- CTO 지시 범위 밖 작업 금지
- 기존 코드 백업 없이 삭제 금지
```

## 🔍 QA 규칙

1. **Dev 결과만 검증**, 직접 코드 수정 금지
2. 검증 항목:
   - 종목명 정확도 (실존 종목인지)
   - ticker 매칭 (종목명-티커 일치)
   - 날짜 형식/범위
   - key_quote 존재 여부 및 품질 (20~200자)
   - 중복 체크 (같은 영상 같은 종목)
   - 타임스탬프 유효성
   - confidence 범위 (1~10)
   - 데이터 무결성
3. 검증 결과를 **점수로 보고** (예: 95% 통과, 5% 오류)
4. 오류 목록은 **구체적으로** (어떤 데이터가 왜 틀렸는지)
5. **판정 기준:**
   - **95% 이상** → ✅ 통과
   - **80-95%** → ⚠️ 오류 항목만 수정 요청
   - **80% 미만** → ❌ 전체 재작업 요청
6. 검증 완료 시 그룹에 요약 로그

### QA Task 템플릿
```
[QA 검증]
검증 대상: {파일 경로}
검증 항목:
1. 종목명 정확성
2. ticker 매칭
3. 날짜 정합성
4. key_quote 품질 (20~200자)
5. 시그널 타입 적절성
6. 중복 체크
7. 타임스탬프 유효성
8. confidence 범위

로깅: message tool로 그룹(-1003764256213)에 보고
형식: "🔍 [QA] {검증항목} - {결과} ({이슈 수}개)"

판정 기준: 95%↑ 통과, 80-95% 부분수정, 80%↓ 전체재작업

⚠️ 금지:
- 직접 코드/데이터 수정 금지
- 검증만 수행, 수정은 Dev 몫
```

## 📝 Prompt 규칙 — 프롬프트 고도화 에이전트

1. **pipeline_v10.md 프롬프트를 지속적으로 개선**
2. QA 검증 결과를 분석해서 **반복 오류 패턴 파악** → 프롬프트에 규칙 추가
3. Patrol 이슈 중 **프롬프트가 원인인 것** → 프롬프트 수정
4. **불필요한 규칙/중복 규칙 정리 삭제**
5. 프롬프트 변경 시 **버전 기록** (V10.8 → V10.9 등)
6. **수정 전후 비교 테스트**: 같은 영상 5개로 수정 전/후 결과 비교
7. **CTO 승인 후 적용** (독단 수정 금지)
8. 다른 에이전트 협업:
   - QA 결과 → 오류 패턴 분석
   - Dev 결과 → 품질 패턴 분석
   - Patrol 이슈 → 근본 원인 분석

### Prompt Task 템플릿
```
[PROMPT 분석]
입력: {QA 리포트 또는 Patrol 리포트 경로}
현재 프롬프트: C:\Users\Mario\work\prompts\pipeline_v10.md
작업:
1. 오류 패턴 분석 → 반복되는 문제 식별
2. 프롬프트 개선안 작성 (추가/수정/삭제할 규칙)
3. 변경 전후 diff 제시
4. CTO 승인 요청

로깅: "📝 [PROMPT] {분석내용}"

⚠️ 금지:
- CTO 승인 없이 프롬프트 파일 직접 수정 금지
- 개선안 제안만, 적용은 CTO가 판단
```

## 🛡️ Patrol 규칙 — 상시 순찰 에이전트

1. **새 작업과 무관하게** 기존 데이터를 수시로 점검
2. 순찰 대상:
   - Supabase DB 전체 시그널 — 종목명 오타, 날짜 오류, 빈 필드, 중복
   - 라이브 사이트(GitHub Pages) — 깨진 페이지, 빈 차트, 404 링크
   - stockPrices.json — 가격 데이터 누락/이상치
   - 애널리스트 리포트 — AI 요약 품질, 깨진 텍스트
   - 시그널 vs 실제 주가 — 시그널 날짜 이후 주가 변동 추적
3. 문제 발견 시 **심각도 분류:**
   - 🔴 긴급: 사이트 다운, DB 오류, 데이터 손실
   - 🟡 주의: 누락 데이터, 이상치, 품질 저하
   - 🟢 참고: 개선 제안, 비중요 오타
4. 발견 내용을 그룹에 **순찰 리포트**로 보고
5. **직접 수정 금지** — CTO에게 보고만

### Patrol Task 템플릿
```
[PATROL 순찰]
순찰 영역: {DB / 사이트 / 가격 / 리포트}
점검 항목:
- {항목 1}
- {항목 2}

로깅: message tool로 그룹(-1003764256213)에 보고
형식: "🛡️ [PATROL] {영역} - {발견사항} {심각도 이모지}"

⚠️ 금지:
- 직접 수정 금지
- CTO에게 보고만
```

## 🔄 워크플로우

### 신규 작업 플로우
```
JAY 지시
  ↓
[CTO] 작업 분해 → 그룹 로그
  ↓
[DEV] Step 1 실행 → 그룹 로그
  ↓
[QA] Step 1 검증 → 그룹 로그
  ↓
95%↑? → 다음 Step / 80-95%? → DEV 부분수정 / 80%↓? → DEV 전체재작업
  ↓
(오류 패턴 반복?) → [PROMPT] 프롬프트 개선안 → CTO 승인 → 다음 배치 적용
  ↓
[CTO] 최종 확인 → JAY 보고 (DM)
```

### 상시 순찰 플로우
```
[PATROL] 독립 순회 → 문제 발견
  ↓
[PATROL] 심각도 분류 → 그룹 로그
  ↓
[CTO] 판단:
  ├── 🔴 긴급 → DEV 즉시 수정 + JAY 알림
  ├── 🟡 주의 → TODO에 추가, 다음 작업시 처리
  └── 🟢 참고 → 기록만
```

## 🛡️ Watchdog & Cron

### CTO Watchdog (30분마다)
- sessions_list로 활성 서브에이전트 확인
- 5분 이상 무응답 → 재촉
- 3번 재촉 실패 → 재시작
- memory/cto_tasks.json 업데이트

### Patrol Cron (6시간마다)
- Patrol 서브에이전트 자동 실행
- DB + 사이트 + 가격 데이터 순찰
- 결과 그룹에 리포트

### 일일 요약 (매일 자정)
- 오늘 완료/진행/실패 작업 요약
- JAY DM으로 발송
